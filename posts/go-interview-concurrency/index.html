Go 并发专项

Posted Apr 7, 2026
By Cc 11 min read

Go 并发专项

高频题（带答）

1. goroutine 与线程的区别？

答： goroutine 由 Go 运行时调度，创建成本低、切换开销小；线程由 OS 调度，创建和切换更重。

2. GMP 模型中 P 的作用？

答： P 维护本地可运行队列并绑定到 M，控制并发度（GOMAXPROCS），是调度核心。

3. GOMAXPROCS 影响什么？

答： 影响同时运行 Go 代码的 P 数量，通常设为 CPU 核数以获得更好的并行度。

4. channel 的无缓冲与有缓冲区别？

答： 无缓冲发送/接收必须同时发生；有缓冲允许异步，直到缓冲满或空。

5. close(channel) 的正确姿势？

答： 仅由发送方关闭；关闭后不能发送，但仍可接收零值且 ok=false。不要重复关闭。

6. select 默认分支作用？

答： 当其它分支都阻塞时执行 default，常用于非阻塞收发或超时控制。

7. nil channel 在 select 中的意义？

答： 对 nil channel 的收发会永久阻塞，因此可以用 nil 禁用 select 分支。

8. WaitGroup 常见错误？

答： 在 goroutine 内 Add 或 Add 与 Wait 并发调用会造成竞态；应先 Add 再启动协程。

9. Mutex 与 RWMutex 选型？

答： 读多写少用 RWMutex，写多或临界区很小用 Mutex 通常更好。

10. sync.Once 的典型场景？

答： 单例初始化、配置加载、连接池初始化等“只执行一次”的逻辑。

11. atomic 与锁的区别？

答： atomic 适合简单数值/指针的无锁更新；锁适合复杂临界区，代码更直观。

12. 如何避免 goroutine 泄漏？

答： 确保所有 goroutine 都有退出路径，使用 context 取消、关闭 channel、超时控制。

13. time.After 高频使用的问题？

答： 频繁创建定时器可能造成资源浪费，建议复用 time.NewTimer。

14. 使用 channel 实现 worker pool？

答： 用任务 channel 分发，worker goroutine 读取任务并处理，结果通过结果 channel 收集。

15. 如何排查并发问题？

答： 使用 go test -race 检测竞态；配合 pprof、trace 分析阻塞与调度。

16. 关闭已关闭的 channel 会怎样？

答： 会 panic。通常只由单一发送方关闭，必要时用 sync.Once 保护。

17. 缓冲 channel 满了会发生什么？

答： 发送方阻塞，直到有接收者取走元素或 channel 被关闭。

18. for range 读取 channel 的终止条件？

答： 当 channel 被关闭且缓冲区数据读完后退出。

19. Go 并发中的“可见性”如何保证？

答： channel 的发送/接收、锁的解锁/加锁形成 happens-before 关系，保证内存可见性。

20. 调度器如何做 work-stealing？

答： 当本地队列为空，P 会从其他 P 的队列尾部偷取 goroutine，提高并行度。

21. Go 的协程抢占是如何实现的？

答： Go 1.14+ 支持异步抢占，可在安全点打断长期运行的 goroutine。

22. select 在多个可执行分支时如何选择？

答： 会随机选择一个分支执行，以避免饥饿。

23. context.WithTimeout 的正确用法？

答： 使用 defer cancel() 及时释放资源，避免 timer 泄漏。

24. fan-in / fan-out 是什么模式？

答： fan-out 用多个 worker 并发处理任务；fan-in 合并多个结果流。

25. 控制并发数量有哪些方式？

答： 缓冲 channel 作为信号量，或使用 x/sync/semaphore。

26. errgroup 的优势？

答： 统一等待多个 goroutine，并能在第一个错误时取消上下文。

27. 数据竞争的典型表现？

答： 并发读写导致非预期结果或 panic，可用 go test -race 检测。

28. RWMutex 可能出现饥饿吗？

答： 读多写少时写者可能饥饿；Go 在特定条件下会进入饥饿模式优先唤醒等待者。

29. Timer 重置的正确姿势？

答： 先 Stop 并清空通道，再 Reset，避免误触发。

30. sync.Cond 的适用场景？

答： 适合“条件满足时唤醒等待者”的场景，比轮询更高效。

深入追问（10题）

31. context 取消与关闭 channel 的区别？

答： context 用于请求级联取消，适合广播式停止；关闭 channel 更适合“数据流结束”的语义。

要点：

  • context 可携带超时与原因。
  • channel 关闭应由发送方控制。

32. 如何发现 goroutine 泄漏？

答： 采集 goroutine profile，定位长期阻塞点；结合日志与指标看 goroutine 数量趋势。

33. atomic.Value 的正确使用方式？

答： 用于并发读多写少的“整体替换”，写入需要保持类型一致。

要点：

  • 适合配置热更新、路由表切换。
  • 不适合局部修改。

34. 并发下如何做“单飞”控制？

答： 使用 singleflight 或者基于 channel 的共享计算，避免相同请求重复计算。

35. channel 缓冲大小如何确定？

答： 依据生产/消费速率与延迟容忍度估算，避免过大造成内存占用，过小导致频繁阻塞。

36. errgroup 的取消传播机制？

答： 任一 goroutine 返回错误会触发 ctx 取消，其它任务需监听 ctx.Done() 及时退出。

37. 如何用信号量控制并发？

答： 使用有缓冲 channel 或 x/sync/semaphore 控制并发数量，获取不到令牌即阻塞或快速失败。

38. RWMutex 的读写饥饿如何缓解？

答： 写多读少时改用 Mutex；读多写少场景避免长期读锁占用。

39. Go 的 netpoller 与并发有什么关系？

答： netpoller 负责网络 IO 事件通知，使 goroutine 在 IO 上阻塞时不占用 OS 线程。

40. worker pool 如何实现背压？

答： 控制任务队列长度，队列满时阻塞或拒绝；必要时增加降级与限流策略。

追问与陷阱（10题）

41. select + default 会带来什么风险？

答： 可能形成忙等（busy loop），导致 CPU 飙高；应加限速或改用阻塞分支。

42. context 超时后仍继续工作的原因？

答： 未在 goroutine 中监听 ctx.Done()；取消只是一种信号，需要显式响应。

43. 如何避免关闭 channel 的竞争？

答： 采用“单一关闭方”原则；多发送者场景用专门协调者或 sync.Once。

44. time.Timer 复用的正确姿势？

答： Stop 后需 drain 通道，再 Reset，避免旧事件误触发。

45. WaitGroup 与 errgroup 如何选？

答： 需要错误聚合/取消就用 errgroup；只需要等待就用 WaitGroup。

46. RWMutex 在读多写少一定更快吗？

答： 不一定，写锁竞争高时反而慢；读锁过多也会导致写饥饿。

47. sync.Map 的读优化机制？

答： 读路径走只读 map，写时进入 dirty map，适合读多写少。

48. atomic 不能替代锁的典型场景？

答： 需要读-改-写的复合逻辑、多变量一致性操作必须用锁。

49. fan-in 合并结果时如何防止阻塞？

答： 使用缓冲 channel 或独立 goroutine 进行合并，并确保有消费者。

50. goroutine 泄漏的常见触发点？

答： 无人接收的发送、无人关闭的接收、无限重试或死循环等待。

package main

import "fmt"

func main() {
    ch := make(chan int, 3)
    for i := 1; i <= 3; i++ {
        ch <- i
    }
    close(ch)

    sum := 0
    for v := range ch {
        sum += v
    }
    fmt.Println(sum)
}
