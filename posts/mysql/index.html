MySQL 面试题

Posted Apr 7, 2026
By Cc 8 min read

MySQL 面试题

高频问题（带答）

1. MySQL 的事务隔离级别？

答： 读未提交、读已提交、可重复读、可串行化。

2. MVCC 解决了什么问题？

答： 通过版本链与 ReadView 解决不可重复读与部分幻读，提高并发。

3. InnoDB 与 MyISAM 区别？

答： InnoDB 支持事务、行锁、外键；MyISAM 不支持事务，使用表锁。

4. 索引类型与使用原则？

答： B+Tree 索引最常用，联合索引遵循最左前缀；避免 LIKE '%x%。

5. EXPLAIN 关注哪些字段？

答： type、key、rows、Extra（是否 using index、using filesort）。

6. binlog/redo/undo 的作用？

答： binlog 用于复制与恢复，redo 用于崩溃恢复，undo 用于回滚与 MVCC。

7. SQL 优化常见点？

答： 避免 select *，减少回表，尽量命中索引，合理分页与联合索引。

8. B+Tree 索引结构的特点？

答： 非叶子节点只存键，叶子节点存数据且有链表，范围查询效率高。

9. 什么是覆盖索引？

答： 查询字段全在索引中，避免回表。

10. 联合索引的最左前缀原则？

答： 复合索引按最左字段开始匹配，跳过中间字段会失效。

11. EXPLAIN 中 type 的常见值？

答： 从好到差：const、eq_ref、ref、range、index、ALL。

12. 索引选择性为何重要？

答： 选择性越高，索引过滤效果越好。

13. InnoDB 锁类型有哪些？

答： 记录锁、间隙锁、临键锁（Next-Key）。

14. 死锁如何处理？

答： InnoDB 自动检测并回滚代价较小的事务；应用侧需重试。

15. binlog 的三种格式？

答： statement、row、mixed；生产常用 row 更安全。

16. 慢查询如何排查？

答： 开启慢查询日志，结合 EXPLAIN 与索引优化。

17. 分页性能问题怎么优化？

答： 用“延迟关联/seek”方式，避免大 offset。

18. 分区表适用场景？

答： 超大表按时间或范围分区，提升维护效率。

19. 读写分离注意点？

答： 注意主从延迟，强一致读需走主库。

20. 复制延迟如何处理？

答： 监控延迟、读主库或降低从库压力。

21. CHARSET/Collation 的影响？

答： 影响排序规则与索引大小，UTF8MB4 兼容更好。

22. TEXT/BLOB 建索引限制？

答： 需指定前缀长度，过长影响性能。

23. UPDATE ... LIMIT 可用吗？

答： 可以，但需谨慎，可能导致不确定性。

24. 主键为何建议自增？

答： 有序插入减少页分裂，提高写入性能。

25. 什么是回表？

答： 索引不覆盖查询字段时，需要根据主键回到聚簇索引取数据。

26. MySQL 如何实现 MVCC？

答： 通过 undo log 与 ReadView 实现多版本并发。

27. redo log 与 undo log 的区别？

答： redo 用于崩溃恢复；undo 用于回滚与 MVCC。

28. explain 的 Extra 常见值？

答： Using index、Using where、Using filesort、Using temporary。

29. 二级索引存什么？

答： 存索引列 + 主键值。

30. 如何防止 SQL 注入？

答： 使用预编译、参数绑定、最小权限原则。

深入追问（10题）

31. InnoDB 与 MyISAM 的主要差异？

答： InnoDB 支持事务、行锁与崩溃恢复；MyISAM 读快但不支持事务与行锁。

32. MVCC 是什么？

答： 多版本并发控制，通过 undo log 维护历史版本，实现非阻塞读。

33. redo log 与 binlog 的区别？

答： redo log 是物理日志，保证崩溃恢复；binlog 是逻辑日志，用于复制与恢复。

34. binlog 的三种格式？

答： statement、row、mixed；row 更安全但体积大。

35. next-key lock 解决什么问题？

答： 防止幻读，在记录锁与间隙锁之间组合。

36. gap lock 的风险？

答： 范围锁容易造成锁冲突与死锁，需要控制范围与索引设计。

37. 覆盖索引的收益？

答： 查询只访问索引不回表，减少 IO，提高性能。

38. ICP（Index Condition Pushdown）作用？

答： 在索引层过滤更多条件，减少回表行数。

39. EXPLAIN ANALYZE 有什么价值？

答： 提供真实执行信息与耗时，帮助验证优化是否生效。

40. 大数据分页的替代方案？

答： 用“游标/seek”方式基于索引定位，避免 OFFSET 扫描。

追问与陷阱（10题）

41. 可重复读能完全避免幻读吗？

答： InnoDB 在可重复读下通过 next-key lock 避免多数幻读，但特定场景仍需注意。

42. SELECT ... FOR UPDATE 会加什么锁？

答： 对命中索引记录加排它锁，范围查询会触发间隙锁。

43. 为什么慢 SQL 经常与索引失效有关？

答： 未命中索引会导致全表扫描，放大 IO 与锁竞争。

44. 覆盖索引不回表的条件？

答： 查询字段完全被索引包含，且查询走该索引。

45. IN 与 EXISTS 如何选？

答： 子查询小且可索引时 IN，大表关联更适合 EXISTS。

46. 为什么避免 SELECT *？

答： 增加 IO 与网络开销，也可能导致索引失效。

47. 索引为什么会“失效”？

答： 隐式类型转换、函数包裹、前导模糊匹配等。

48. 自增主键一定好吗？

答： 顺序写友好，但在分布式场景可能成为瓶颈，需要全局 ID 方案。

49. 事务长时间不提交的危害？

答： 占用锁与 undo 空间，阻塞其他事务。

50. 数据库连接池为什么会“打满”？

答： 慢 SQL、连接泄漏或并发激增，需要限制池大小与超时。
