系统设计专项

Posted Apr 7, 2026
By Cc 11 min read

系统设计专项

高频题（带答）

1. 什么是高可用？

答： 通过冗余、故障转移、健康检查保证服务在部分故障时仍可对外提供服务。

2. 如何做限流？

答： 常见算法：固定窗口、滑动窗口、令牌桶、漏桶。落地可用 Redis 或本地内存实现。

3. 如何做熔断与降级？

答： 熔断在失败率高时快速失败并保护下游；降级提供兜底方案（默认值/缓存/静态页）。

4. 设计一个短链系统的关键点？

答： ID 生成（自增/雪花/哈希）、去重、缓存、过期、统计、黑名单与反爬。

5. 如何设计消息队列消费幂等？

答： 通过业务唯一键去重，或基于状态机、数据库唯一约束实现幂等。

6. 如何解决分布式事务？

答： 常见方案：本地消息表+最终一致性、TCC、SAGA；实际选型视业务复杂度。

7. 设计一个登录系统需要考虑什么？

答： 认证方式、密码加密、会话管理、过期策略、多端登录与安全审计。

8. 分库分表如何选路由键？

答： 选择高基数且均匀分布的字段（如 user_id），避免热点与跨分片查询。

9. 如何做缓存一致性？

答： 常用写策略：先写库再删缓存或写缓存；结合过期时间与双删策略降低不一致窗口。

10. 如何做可观测性？

答： 三板斧：日志、指标、链路追踪；统一 TraceID 贯穿请求链路。

11. CAP 定理是什么？

答： 一致性、可用性、分区容错性三者不可同时满足，分布式系统必须权衡。

12. 什么是最终一致性？

答： 允许短时间不一致，最终通过重试/补偿达到一致。

13. 缓存模式 Cache-Aside 的读写流程？

答： 读：先查缓存，未命中读库再回填缓存；写：先写库再删缓存。

14. 限流放在网关还是服务内？

答： 网关统一入口更易控流；服务内限流适合局部保护，通常两者结合。

15. 熔断器的三态？

答： 关闭、打开、半开；半开用于试探下游是否恢复。

16. 如何保证消息有序？

答： 分区内有序，按 key 分区；全局有序需单分区或顺序队列，吞吐会下降。

17. ID 生成方案有哪些？

答： 自增、UUID、雪花算法、号段分配；需考虑有序性与冲突。

18. 分库分表如何迁移？

答： 双写/灰度迁移，配合数据校验与回滚方案。

19. 热点 key 怎么处理？

答： 加随机后缀分散、拆分、缓存预热、局部降级。

20. 多活与容灾的区别？

答： 多活是多机房同时承载流量；容灾强调故障切换。

21. 蓝绿与金丝雀发布区别？

答： 蓝绿是两套环境切流；金丝雀是小流量逐步放量。

22. SLA/SLI/SLO 的关系？

答： SLI 是指标，SLO 是目标，SLA 是对外承诺。

23. 如何做全链路追踪？

答： 统一 TraceID、SpanID，通过中间件传递并上报。

24. 如何设计任务调度系统？

答： 任务元数据、调度器、执行器、重试与失败告警。

25. 大文件上传如何设计？

答： 分片上传、断点续传、校验与合并。

26. 搜索系统的核心数据结构？

答： 倒排索引；支持分词、召回与排序。

27. push 与 pull 的适用场景？

答： push 适合实时性高；pull 适合客户端控制节奏。

28. 配置中心的关键能力？

答： 版本管理、灰度发布、监听推送、审计。

29. 认证与授权的区别？

答： 认证确认“你是谁”，授权确认“你能做什么”。

30. 可观测性三件套是什么？

答： 日志、指标、链路追踪。

深入追问（10题）

31. CAP 定理在实际系统中的取舍？

答： 面对网络分区时只能在一致性与可用性中二选一；多数系统选择 AP 或 CP，并通过业务补偿折中。

32. 一致性模型有哪些常见类型？

答： 强一致、线性一致、因果一致、最终一致；一致性越强，延迟与可用性代价越高。

33. 一致性哈希解决了什么问题？

答： 节点扩缩容时减少数据迁移量，适合缓存与分片路由。

34. 分布式 ID 生成常见方案？

答： 雪花算法、号段服务、数据库自增；需权衡唯一性、单调性与可用性。

35. 读写分离的注意点？

答： 需要解决主从延迟导致的读到旧数据，可通过读主库、读写一致性标签或延迟读。

36. Cache Aside 的一致性问题？

答： 写库与删缓存顺序不当会导致脏读，常用“先删缓存再写库 + 延迟再删”。

37. 消息队列的顺序性如何保证？

答： 通过单分区/单消费者保证顺序；跨分区需按业务 key 进行分区。

38. 熔断（Circuit Breaker）的触发条件？

答： 以错误率、超时率为阈值；触发后进入半开探测，恢复后关闭熔断。

39. SLA/SLO/SLI 的区别？

答： SLI 是指标，SLO 是目标，SLA 是对外承诺与赔付条款。

40. 蓝绿/金丝雀发布的差异？

答： 蓝绿是全量切换；金丝雀是小流量灰度，风险更低但需要完善监控。

追问与陷阱（10题）

41. 幂等接口如何落地？

答： 幂等键 + 去重表/状态机，确保重复请求不产生副作用。

42. “先删缓存再写库”一定安全吗？

答： 存在并发回源写入旧值风险，通常用“先写库再删缓存 + 延迟再删”。

43. 为什么需要熔断与降级一起设计？

答： 熔断隔离故障源，降级保证核心链路可用，组合更稳。

44. 限流阈值如何定？

答： 根据 SLO 与容量评估，结合压测结果与业务高峰流量。

45. 数据分片后全局排序怎么做？

答： 通过全局索引、归并排序或搜索引擎辅助。

46. 分布式事务的优先原则？

答： 能不做就不做，优先通过业务拆分、异步化与补偿设计。

47. 配置灰度发布的核心保障？

答： 版本隔离、回滚机制、完整审计与观测。

48. 多活架构的挑战？

答： 数据一致性、冲突解决、跨地域延迟。

49. 灰度期间如何观测质量？

答： 对比健康指标、错误率、延迟分布，设置自动回滚阈值。

50. 分布式缓存一致性有哪些策略？

答： Cache Aside + 延迟双删，或 CDC 驱动的事件刷新。

package main

import "fmt"

func main() {
    requests := []string{"a", "a", "b", "b"}
    seen := map[string]bool{}

    for _, id := range requests {
        if seen[id] {
            fmt.Println("dup", id)
            continue
        }
        seen[id] = true
        fmt.Println("ok", id)
    }
}
