By Cc 6 min read

Go进阶

速查要点

  • 并发同步：Mutex、RWMutex、WaitGroup、Once
  • channel 与 select 的细节
  • context 的取消传播
  • 错误包装与判断
  • map 实现与并发问题
  • 逃逸分析与内存分配

常见面试题（带答）

1. map 是否并发安全？

答： 不是。并发读写会 panic。可用 sync.Mutex/RWMutex 或 sync.Map。

2. sync.Map 适用场景？

答： 读多写少、key 稳定且长期存在的场景。写多时普通 map+锁 通常更好。

3. select + nil channel 的效果？

答： 对 nil channel 的发送/接收会永久阻塞，在 select 中相当于禁用该分支。

4. close(channel) 的规则？

答： 只能由发送方关闭；关闭后不能再发送，但可以继续接收（会返回零值且 ok=false）。

5. WaitGroup 的正确用法？

答： Add 要在启动 goroutine 之前调用，Done 保证执行一次；不要在 goroutine 内部对同一个 WaitGroup 再 Add。

6. context 如何取消传播？

答： 上游使用 WithCancel/WithTimeout 创建子上下文，下游在逻辑中监听 ctx.Done()。

7. defer 的坑？

答： 循环中大量 defer 会堆积直到函数返回，可能造成内存增长。

8. error 包装与判断如何做？

答： 用 fmt.Errorf("...: %w", err) 包装，用 errors.Is/As 判断。

9. 什么情况下会逃逸到堆？

答： 变量生命周期超出函数范围（返回局部变量指针、闭包捕获）可能逃逸。可用 go build -gcflags="-m" 查看。

10. time.After 的注意点？

答： 高频调用会产生大量定时器对象，建议复用 time.NewTimer 并 Reset()。

11. context 常见使用错误？

答： 忘记取消导致泄漏、在 goroutine 内丢失 ctx、把 ctx 存到结构体长期持有。

12. sync.Pool 的适用场景？

答： 高频临时对象复用（如 buffer），但不保证一定命中，不能存放必须存在的对象。

13. defer 与 panic 的执行顺序？

答： 发生 panic 时仍会执行所有 defer，执行顺序仍是 LIFO。

14. map 的删除与遍历？

答： delete 在遍历中是安全的，但遍历顺序不确定，结果不可依赖。

15. 接口值为 nil 的陷阱是什么？

答： 接口由“动态类型 + 动态值”组成，动态值为 nil 但动态类型非 nil 时，接口不等于 nil。

要点：

  • var err error = (*MyErr)(nil) 不等于 nil。
  • 需显式返回 nil 接口。

16. context.Value 的正确用途？

答： 只用于请求范围的元数据（trace id、user id），不应作为业务参数或可选配置传递。

17. errgroup 的优势与注意点？

答： 统一等待 goroutine 并在首个错误时取消；注意每个任务要尊重 ctx.Done()。

18. sync.Cond 什么时候用？

答： 当多个 goroutine 等待同一条件且需要“唤醒”机制时，Cond 比轮询更高效。

19. 用原子操作替代锁的条件？

答： 仅在操作非常简单（计数、指针交换）且无复合逻辑时使用原子；复杂临界区仍应使用锁。

20. close(channel) 的“拥有者”原则？

答： 谁负责发送，谁关闭；接收方不关闭。多发送者需通过协调（单独管理者、sync.Once）统一关闭。

21. pipeline 如何优雅退出？

答： 使用 context 或 done channel 通知下游停止，避免 goroutine 泄漏。

22. time.Ticker 的常见泄漏原因？

答： 不调用 Stop()；持续创建新的 ticker 但未释放。

23. select 超时的标准写法？

答： 使用 time.After 或复用 time.Timer，并在超时分支里返回或清理资源。

24. 并发读写 map 的正确方式？

答： 读多写少用 RWMutex；读写频繁且 key 稳定可用 sync.Map；更极端的读场景可做分片 map。

25. defer 在循环里的性能影响？

答： 大量 defer 会堆积直到函数返回，增加内存占用和执行开销，应改为显式调用或封装函数。

26. panic 如何做到可控恢复？

答： 在边界层（如 goroutine 入口、HTTP handler 中间件）统一 recover，记录日志并转换为错误。

27. Go 的逃逸分析如何查看？

答： 使用 go build -gcflags='-m 或 go test -gcflags='-m 查看逃逸原因。

28. sync.Pool 为什么不适合缓存长期对象？

答： pool 会在 GC 时被清理，且命中不保证，适合临时对象复用而非缓存。

29. map 迭代时能否写入？

答： 不能并发写入；单协程内边迭代边写入也可能 panic，应先收集 key 再写。

30. 泛型的约束（constraint）怎么理解？

答： 通过接口约束类型集合，决定可用的运算与方法集合，是泛型可用性与安全性的核心。
