高并发架构专题

高频题（带答）

1. QPS、TPS 与并发数的关系？

答： QPS/TPS 是吞吐量，和并发、响应时间相关：吞吐 ≈ 并发 / 平均响应时间。

2. 高并发系统的核心目标？

答： 保证稳定性与可扩展性，同时在可控成本下提升吞吐与降低延迟。

3. 负载均衡的常见策略？

答： 轮询、最少连接、权重、hash、一致性哈希。

4. 多级缓存的价值？

答： 降低回源压力，提升热点数据访问速度。

5. CDN 在高并发中的作用？

答： 静态内容就近分发，显著降低源站压力与带宽消耗。

6. 读写分离的优势与风险？

答： 提升读吞吐，但需处理主从延迟导致的一致性问题。

7. 分库分表适用场景？

答： 单库性能瓶颈明显或数据量过大时，需水平拆分。

8. 限流的常用算法？

答： 令牌桶、漏桶、固定窗口、滑动窗口。

9. 熔断与降级的区别？

答： 熔断隔离故障服务；降级提供低配服务保证可用。

10. 削峰填谷的常用手段？

答： 消息队列、异步化、批处理。

11. 为什么要做异步化？

答： 将非关键路径任务移出主流程，提高响应速度与系统吞吐。

12. 连接池的关键参数？

答： 最大连接数、空闲连接数、连接超时、最大生命周期。

13. 数据库索引优化在高并发中的作用？

答： 降低 IO 与锁竞争，提高单机吞吐上限。

14. 慢 SQL 的治理思路？

答： 分析执行计划、补充索引、改写查询、减少回表。

15. 热点数据的隔离方法？

答： 本地缓存、分片、热点路由、限流。

16. 请求级别的幂等设计？

答： 使用幂等键、状态机、去重表确保重复请求结果一致。

17. 全链路压测的意义？

答： 发现瓶颈与容量上限，验证限流/降级是否生效。

18. 弹性扩缩容的触发指标？

答： CPU、QPS、延迟、队列堆积、错误率等。

19. Backpressure 在系统中的位置？

答： 在入口或关键中间层限制流量，防止级联故障。

20. 服务拆分的优缺点？

答： 提升自治与扩展，但引入分布式复杂性与网络开销。

21. 限流与排队如何选择？

答： 实时性强用限流；可延迟处理的请求可排队。

22. 读多写少的优化组合？

答： 多级缓存 + 只读副本 + 预热。

23. 大促场景的稳定性策略？

答： 限流、降级、熔断、动态扩容、预案演练。

24. 资源隔离（Bulkhead）是什么意思？

答： 将资源分隔在独立池中，避免单点过载拖垮全局。

25. 灰度发布在高并发中的作用？

答： 小流量验证新版本稳定性，避免全量故障。

26. 配置中心的价值？

答： 动态开关、灰度策略与快速回滚能力。

27. 监控体系的关键指标？

答： 延迟、吞吐、错误率、饱和度（四黄金指标）。

28. SLA/SLO 在容量规划中的作用？

答： 以目标可用性与延迟反推资源与冗余需求。

29. 事故恢复的常见策略？

答： 回滚、降级、切流、容灾切换。

30. 容量规划的核心输入？

答： 峰值流量、增长预期、业务模型与可接受的成本。

追问与陷阱（10题）

31. 为什么高并发下“长尾延迟”更明显？

答： 资源竞争、队列堆积与 GC/抖动会放大尾部延迟。

32. 连接池太大有什么风险？

答： 增加数据库负载与上下文切换，导致整体吞吐下降。

33. 限流后仍被打爆的原因？

答： 限流设置过高、入口未统一、下游服务未隔离。

34. 降级策略如何设计优先级？

答： 按业务重要性与用户影响程度设置“可降级清单”。

35. 热点隔离为什么要结合路由？

答： 仅缓存不足以隔离热点，需路由到专用资源池。

36. 为什么要做“容量水位线”？

答： 提前触发限流/扩容，避免突增流量直接击穿。

37. 伸缩策略的常见误区？

答： 只看 CPU，不看延迟与队列堆积，会导致扩容滞后。

38. 如何避免级联失败？

答： 熔断、隔离、超时、重试退避与降级配合。

39. 高并发系统的测试重点？

答： 峰值压力、稳定性、限流降级生效与恢复能力。

40. 全链路压测的关键前提？

答： 流量隔离与真实业务模型，避免影响线上用户。

package main

import (
    "fmt"
    "math"
)

func main() {
    qps := 1200.0
    latency := 0.2 // 200ms
    perInstanceConcurrency := 50.0

    need := math.Ceil(qps * latency / perInstanceConcurrency)
    fmt.Println(int(need))
}
