Redis 面试题

Redis 面试题

高频问题（带答）

1. Redis 常用数据结构及场景？

答：

  • string：计数、验证码、配置缓存
  • hash：对象属性（用户信息）
  • list：消息队列、时间线
  • set：去重、关系集合
  • zset：排行榜、延迟队列

2. Redis 持久化方式？

答： RDB（快照）与 AOF（追加日志）。RDB 恢复快但可能丢数据；AOF 更完整但体积更大。

3. 缓存雪崩/击穿/穿透？

答：

  • 雪崩：大量 key 同时失效，解决方式是过期时间加随机。
  • 击穿：热点 key 失效，解决方式是互斥锁/逻辑过期。
  • 穿透：请求不存在的数据，解决方式是布隆过滤器或缓存空值。

4. Redis 的过期淘汰策略？

答： 常见策略有 volatile-lru、allkeys-lru、volatile-ttl、noeviction 等。

5. 如何实现分布式锁？

答： 使用 SET key value NX EX seconds，并用 Lua 脚本保证解锁时 value 匹配。

6. Redis 为什么快？

答： 内存存储、单线程事件循环、IO 多路复用、数据结构优化。

7. Pipeline 与事务的区别？

答： Pipeline 是减少 RTT 批量发送，不保证原子性；事务通过 MULTI/EXEC 保证原子执行。

8. Redis 的主从复制原理？

答： 主节点将写命令异步同步给从节点，故障时可通过哨兵或集群完成切换。

9. Sentinel 与 Cluster 的区别？

答： Sentinel 负责高可用与故障切换；Cluster 解决分片与水平扩展。

10. Redis Cluster 的槽位数量？

答： 16384 个 hash slot，key 通过 CRC16 映射到槽位。

11. Lua 脚本有什么特点？

答： 保证原子性，减少网络往返，但脚本执行过久会阻塞。

12. Pipeline 与 Lua 的差异？

答： Pipeline 仅减少 RTT；Lua 保证原子执行。

13. AOF 重写的意义？

答： 压缩 AOF 文件，降低恢复时间与空间占用。

14. big key 的危害？

答： 阻塞主线程、导致延迟抖动；应拆分或使用 scan。

15. hot key 的处理？

答： 本地缓存、分片、限流、提前预热。

16. Redis Cluster 迁移如何进行？

答： 通过 reshard 迁移 slot，迁移期间会产生重定向。

17. WATCH 的作用？

答： 乐观锁，监视 key 变化；变化则 EXEC 失败。

18. Pub/Sub 的特点？

答： 实时广播、不持久化、无回溯。

19. Stream 适用场景？

答： 消息队列与事件流，支持消费组与 ACK。

20. Bitmap 的应用？

答： 用户签到、在线状态、基数统计。

21. HyperLogLog 的特点？

答： 近似去重计数，误差可接受且空间小。

22. GEO 功能场景？

答： 附近的人、位置检索。

23. Redis 过期 key 删除策略？

答： 定期删除 + 惰性删除结合。

24. Redis 慢查询如何排查？

答： 使用 SLOWLOG，结合 INFO 和 latency 指标。

25. RDB 触发方式？

答： save/bgsave 或配置定期快照。

26. maxmemory-policy 如何选？

答： 读多选择 allkeys-lru，写多结合业务容忍度。

27. 分布式锁的坑？

答： 时钟漂移、续期、误删；需要唯一 value 校验与续约。

28. Redis 与 DB 双写一致性？

答： 常用“写库后删缓存”，配合延迟双删或消息队列。

29. MGET/HMGET 的意义？

答： 批量获取减少 RTT，提高性能。

30. 为什么 Redis 单线程还能高性能？

答： 事件循环 + IO 多路复用，内存操作快且避免锁竞争。

深入追问（10题）

31. Redis 真的“单线程”吗？

答： 命令执行是单线程事件循环，但 IO、持久化与后台任务是多线程/多进程。

32. AOF 的 fsync 策略有哪些？

答： always 每次刷盘最安全最慢；everysec 性能与安全折中；no 由 OS 决定。

33. Redis 启动时优先加载 RDB 还是 AOF？

答： 若开启 AOF 且文件存在，优先加载 AOF；否则加载 RDB。

34. 主从复制延迟如何应对？

答： 读写一致性要求高时读主库；或通过“读写一致性标记”做延迟读控制。

35. Sentinel 的故障转移流程？

答： 选主、主观下线/客观下线判断、投票选举、提升从库并通知客户端更新。

36. Cluster 的 hash tag 有什么用？

答： 让多个 key 落到同一 slot，便于多 key 操作。

37. 内存碎片率过高怎么办？

答： 优化 key 粒度、避免大 key；必要时使用 MEMORY PURGE 或滚动重启。

38. Keyspace Notifications 的用途？

答： 订阅 key 过期、删除事件，用于缓存同步或失效通知。

39. 如何识别 hot key？

答： 结合 monitor/slowlog/业务侧统计，或用代理侧采样。

40. 分布式锁常见坑？

答： 锁过期导致并发进入、解锁误删；需设置唯一 value 并用 Lua 脚本校验解锁。

追问与陷阱（10题）

41. Redis Cluster 跨槽操作为什么会失败？

答： 多 key 不在同一 slot 时无法保证原子性，需要使用 hash tag。

42. SCAN 相比 KEYS 的优势？

答： SCAN 逐步遍历避免阻塞主线程，适合线上使用。

43. big key 如何治理？

答： 拆分 key、分桶存储、使用集合分片。

44. hot key 如何识别？

答： 业务侧统计 + slowlog + 代理采样，结合 QPS 分布分析。

45. 为什么 Lua 脚本要控制执行时间？

答： 脚本执行期间会阻塞事件循环，影响整体延迟。

46. Redis 复制的 PSYNC 有何作用？

答： 增量复制，减少全量同步成本。

47. AOF 重写为什么是后台执行？

答： 避免阻塞主线程，保证服务可用性。

48. MEMORY PURGE 能解决什么问题？

答： 释放 jemalloc 碎片，但会带来短时抖动。

49. 为什么 Redis 不适合作为消息队列的唯一方案？

答： Pub/Sub 不持久化，Stream 需要严格消费与监控保证。

50. Redis 事务 MULTI/EXEC 有哪些限制？

答： 不支持回滚，命令入队阶段只做语法检查。
